# WF-12-临时新增 审批工作流实施方案 v1.0

## 框架定位

本文档作为 NocoBase 平台同类审批工作流的标准实施框架。后续其他业务（如合同审批、采购审批等）可参照此结构复制。

## 需求概览

| # | 需求 | 实现方式 |
|---|------|---------|
| 1 | 按预计造价三路分支审批：<500→项目经理，>2000→分管领导，500~2000→部门负责人 | Condition 节点嵌套 |
| 2 | 审批后更新审批意见 + 记录审批人昵称 | Script 节点操作 DB |
| 3 | 更新工作流状态 | Script 节点操作 DB |
| 4 | 通知申请人 | Script 节点写入 notificationInAppMessages |
| 5 | 审批通过生成报备简报 | Script 节点写入 briefings（`briefing_type: "报备简报"`）|

## 节点拓扑

```
[0] TRIGGER: collection → target_table [mode=0, appends: relation_field]
    │
[1] QUERY: 查询当前记录 [appends: relation_field]
    │
[2] CONDITION: 条件A (如 cost < 500?)
    ├─TRUE───────────── FALSE ─────────────┐
    │                                       │
[3a] SCRIPT: 查找审批人A         [4] CONDITION: 条件B (如 cost > 2000?)
    │  按规则设置 approver_id    ├─TRUE────── FALSE ────┐
    │                            │                     │
[5a] MANUAL: 审批节点A    [3b] SCRIPT: 查审批人B  [3c] SCRIPT: 查审批人C
    │  assignees:[approver_id]  │                     │
    │  forms:{意见,结果}        │                     │
    ↓                           ↓                     ↓
[5b] MANUAL: 审批节点B          [5c] MANUAL: 审批节点C
    ↓                           ↓                     │
    └──────────────┬────────────┴─────────────────────┘
                   ↓ (合并点: 所有 Manual downstream 指向同一节点)
[6] SCRIPT: 审批结果处理
    │  读取 context.approval → 更新 DB
    │  生成简报HTML → 写入 briefings
    │  写入通知 → notificationInAppMessages
    │
[7] CONDITION: 审批通过? (保留供工作流历史标记)
    ├─TRUE → 结束
    └─FALSE → 结束
```

### 核心原则

1. **三路互斥**：Condition 嵌套实现，任一分支执行后即结束，不逐级审批
2. **合并点**：所有 Manual 节点的 downstream 指向同一个 Script 节点
3. **不另设 Create 节点**：简报和通知都在 Script 节点中用 `context.db` 直接操作
4. **审批人递归查找**：本级部门无对应角色，则递归父部门

## 标准脚本模板

### 3a/3b/3c：审批人查找（模板）

```javascript
async function findUpward(deptId, db, flag) {
  var du = await db.getRepository('departmentsUsers').findOne({
    filter: { departmentId: deptId, [flag]: true }
  });
  if (du) return du.userId;
  var dept = await db.getRepository('departments').findOne({
    filter: { id: deptId }
  });
  if (dept && dept.parentId) return await findUpward(dept.parentId, db, flag);
  return null;
}

async function main(context) {
  var record = context.data;
  var project = await context.db.getRepository('projects').findOne({
    filter: { id: record.project_id }
  });
  if (!project || !project.departmentId) return false;

  var approverId = await findUpward(project.departmentId, context.db, 'FLAG_HERE');
  if (!approverId) return false;

  context.data.approver_id = approverId;
  return true;
}
```

### 6：审批结果处理（模板）

```javascript
async function main(context) {
  var approval = context.approval || {};
  var status = approval.status || 'approved';
  var formValues = approval.values || {};
  var record = context.data;
  var currentUserId = context.execution.context.userId;

  var approver = currentUserId ? await context.db.getRepository('users').findOne({
    filter: { id: currentUserId }
  }) : null;
  var approverNickname = (approver && approver.nickname) || '';

  var newStatus = (status === 'approved') ? '已审批' : '已驳回';
  var opinion = formValues.approver_opinion || (status === 'approved' ? '同意' : '驳回');

  // 1. 更新业务表
  await context.db.getRepository('TARGET_TABLE').update({
    filter: { id: record.id },
    values: {
      status: newStatus,
      approver_opinion: opinion,
      approver_id: currentUserId,
      approver_nickname: approverNickname
    }
  });

  // 2. 生成简报 HTML（参照 bc-card 格式）
  // ... 业务字段映射 ...

  // 3. 创建简报记录（仅在审批通过时）
  if (status === 'approved') {
    await context.db.getRepository('briefings').create({
      values: {
        briefing_type: 'BRIEFING_TYPE',
        title: 'TITLE',
        summary: html,
        briefing_date: new Date(),
        project_id: record.project_id,
        source_workflow_id: record.id
      }
    });
  }

  // 4. 通知申请人
  if (record.applicant_id) {
    await context.db.getRepository('notificationInAppMessages').create({
      values: {
        userId: record.applicant_id,
        title: 'NOTIFICATION_TITLE',
        content: 'NOTIFICATION_CONTENT',
        status: 'sent'
      }
    });
  }

  return (status === 'approved');
}
```

## WF-12 具体实施

### 数据表变更

| 表 | 变更 | 说明 |
|----|------|------|
| `temporary_additions` | 新增字段 `approver_nickname` (string) | 存储审批人昵称 |
| `briefings` | `briefing_type` 枚举增加 `"报备简报"` | 简报类型 |

### 脚本清单（4个，内嵌到 Script 节点）

| 节点 | 脚本用途 | 查找逻辑 |
|------|---------|---------|
| 3a | 查找项目经理审批人 | 子部门 isOwner → project_manager 角色 |
| 3b | 查找分管领导审批人 | 部门 is_manager_in_charge → 递归父部门 |
| 3c | 查找部门负责人审批人 | 部门 isOwner → 递归父部门 |
| 6 | 审批结果处理 + 简报 + 通知 | 见上节 |

### 不修改的文件

| 文件 | 原因 |
|------|------|
| `scripts/briefing-worker.js` | 施工日志轮询服务，隔离 |
| `briefing-shell.html` | 前端简报展示，自动支持新增类型 |
| `workflow-scripts/briefing-template.js` | 施工日志简报模板 |
| 任何 `TIMER-*`, `SERIAL-*` 文件 | 其他工作流，隔离 |

### 实施步骤（NocoBase UI）

1. **集合设置**：配置 `approver_nickname` 字段，`briefing_type` 枚举
2. **Trigger**：修改 appends `["project_id_id"]`
3. **Query 节点**：修改 appends `["project_id_id"]`
4. **删除**：原"部门审批"和"分管领导审批"两个 Manual 节点
5. **新建 Condition**：`cost < 500?`
6. **新建 Script**：查找项目经理审批人
7. **新建 Manual**：项目经理审批（`assignees: ["{{$context.data.approver_id}}"]`）
8. **新建 Condition**：`cost > 2000?`（接在 cost<500 的 false 分支）
9. **新建 Script**：查找分管领导审批人
10. **新建 Manual**：分管领导审批
11. **新建 Script**：查找部门负责人审批人
12. **新建 Manual**：部门负责人审批
13. **修改 Update**：补充 values（status/approver_opinion/approver_id）
14. **新建 Script**：审批结果处理（合并点，含简报+通知）
15. **连接**：确保 4a/4b/4c → 6，6 → 7，7 → 结束
16. **测试**：三种造价分别测试

## 同类方案复用指南

当建立新的审批工作流时：

1. 复制本拓扑结构
2. 替换 `target_table` 为业务表
3. 调整 Condition 分支条件
4. 替换 3a/3b/3c 审批人查找逻辑（角色/标志位）
5. 替换 6 中的简报字段映射和通知模板
6. 保持合并点、通知、简报创建模式不变
